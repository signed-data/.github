# SignedData.Org

---

## The problem first appeared in a barber shop

I was building an AI-powered SaaS and one of the requirements was to keep a conversation with the customers, based on details that "the Barber" used to describe that peculiar client on the system.

Customized chats to engage the customer near the date we predict he will probably return. It worked well, but I had the idea to use real, factual data to engage these conversations.

This way I would solve two problems. Some platforms are rigid about letting models fill the content of the message, and I wanted to do more than just small talk. With factual data, models can ground the conversation at a very confident level.

---

## The gap has a name

Large language models are extraordinarily good at reasoning. They are not good at knowing whether the premises of their reasoning are true.

This is not a flaw in the models. It is a flaw in how data reaches them.

When you hand an AI a JSON payload from an API, you are handing it a string of bytes. Those bytes might be correct. They might have been altered in transit. They might be cached from three days ago. They might come from a source that no longer exists. The model has no way to know. It reasons from the data as given, as if it were fact, and produces outputs that downstream systems and human beings will act on.

For entertainment, this is fine. For financial decisions, compliance verification, business intelligence, legal research — it is not.

The open data movement spent a decade making government and institutional data accessible. Tim Berners-Lee gave us the four rules of Linked Data in 2006: name things with URIs, make those URIs dereferenceable, return useful information, link to other things. The semantic web community built vocabularies and ontologies. Hundreds of public APIs opened up.

But nobody built the trust layer.

Data was open. Data was structured. Data was sometimes linked. Data was almost never *signed*.

---

## What we built

SignedData is an open standard for wrapping any real-time data source in a universal signed envelope.

Every event carries:

- A **cryptographic signature** — RSA-PSS SHA-256, computed over the canonical bytes of the event's **signed payload**. Any consumer with the issuer's public key can verify it independently, with no network call, no trusted intermediary.
- A **fingerprint** of the raw upstream API response — SHA-256 of the exact bytes returned by the source, before any parsing or normalisation.
- A **dereferenceable identity** — the source, the content type, the issuer are all HTTP URIs that you can look up and find a real document. Not opaque strings. Links.
- A **context summary** — a human-readable, language-tagged description generated by a lightweight model, so that data and meaning travel together. It is interpretive metadata carried alongside the event and is **not** part of the signed payload.

We built it on Tim Berners-Lee's four rules because they are correct. A `CDSEvent` is valid JSON-LD. Parse it with any RDF processor and it expands to proper triples. Follow the links and you arrive at the source registry, the vocabulary, the public key. The chain is complete and navigable.

The signing algorithm does not change what the data says. It proves who said it, when they said it, and that nobody changed it since.

---

## The standard is not the product

This is important to understand about what SignedData is and is not.

The Curated Data Standard is MIT-licensed and infrastructure-free. The spec, the Python SDK, the TypeScript SDK, the vocabulary — they belong to the community. Anyone can run their own ingestors, sign with their own key, and publish CDS events with their own `signed_by` identity. The trust anchor is their organisation, not ours.

We publish the reference implementation and run the reference deployment at `signed-data.org`. We are the first operator of a standard we hope others will adopt.

This is the same model as HTML and Apache, Kubernetes and EKS, SMTP and Gmail. The W3C defines HTML. You run the web server. We define CDS. You ingest the data.

## What we curate

Today, `signed-data.org` produces signed feeds for:

- **Brazilian lotteries** — Mega Sena, Lotofácil, Quina, Lotomania, Dupla Sena. Official Caixa Econômica Federal data, signed within seconds of each draw.
- **Football results** — Brasileirão, Libertadores, major European leagues. Match results, live scores, standings.
- **Weather** — Current conditions and forecasts for Brazilian and international cities via Open-Meteo.
- **Financial rates** — SELIC, IPCA, PTAX, CDI from Banco Central do Brasil. B3 stock quotes.
- **Company registry** — CNPJ lookups from the Receita Federal via BrasilAPI. Signed at the moment of query.

Every feed is exposed as an MCP server — tools that Claude and other LLMs can call directly, receiving data that carries a cryptographic proof of its origin.

The model is what the open data literature calls an *infomediary*: we sit between the original source and the end consumer, adding structure, signing, and semantic enrichment without altering the underlying facts.

## Who this is for

If you are building AI applications that reason about real-world facts — prices, regulations, company information, sports results, public records — you have a trust problem you may not have named yet.

SignedData is for developers and organisations who want to eliminate that trust problem at the source, before data enters any reasoning pipeline.

It is for compliance teams who need an auditable chain from a business decision back to the exact data the AI used to make it.

It is for engineers who are tired of asking "how do I know this is right?" and getting no answer.

It is for anyone who believes that if an AI is going to act on a fact, someone should have signed that fact.

## Built in Brazil, designed for the world

SignedData was started in Jundiaí, São Paulo, by a software engineer with two decades of experience building systems that handle consequential data — financial records, customer interactions, production pipelines.

Brazil is an interesting place to start. The country has exceptional public data infrastructure — Banco Central's APIs, IBGE's datasets, the Receita Federal's CNPJ registry, the Caixa lottery system — much of it free, structured, and legally redistributable. It is also a country where the distance between official data and what actually reaches systems is wide enough to matter. The trust problem is visible here.

But the problem is not Brazilian. Every LLM deployment in every country that reasons about real-world data faces the same gap. The standard is designed to be universal — the vocabulary, the signing algorithm, the Linked Data architecture are all language- and jurisdiction-neutral. The Brazilian feeds are where we started. They are not where we end.

## The principle we will not compromise

Hallucination is a hard error.

When SignedData signs an event, we are attesting that a specific source returned specific bytes at a specific time. We are not attesting that the source is correct. We are not predicting what will happen next. We are not summarising or interpreting beyond what the data says.

The `context.summary` field in every CDS event is explicitly labeled with the model that generated it. It is a convenience for human readers, not a fact claim. The signature does not cover interpretations. It covers facts.

If a tool built on SignedData produces an output that cannot be traced to a signed source event, it must say so. The label `"llm_generated": true` is not a warning. It is a mandatory disclosure.

We built this because we believe that the value of AI in consequential decisions depends entirely on the quality of the facts it reasons from. And facts, to be trusted, must be signed.
